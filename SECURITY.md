# Политика безопасности

## Поддерживаемые версии

| Версия | Статус поддержки |
| ------ | ---------------- |
| 5.1.x  | ✅ Обновления безопасности |
| 5.0.x  | ❌ Не поддерживается |
| 4.0.x  | ✅ Ограниченные исправления |
| < 4.0  | ❌ Не поддерживается |

## Ответственное раскрытие

Пожалуйста, не публикуйте детали уязвимости публично до её исправления.

### Как сообщить
1. Отправьте письмо на: nerdlinnut@gmail.com.
2. Тема: SECURITY REPORT: краткое описание.
3. Укажите:
   - Версию проекта / commit hash
   - Шаги воспроизведения
   - Ожидаемое vs фактическое поведение
   - Оценку воздействия (данные, эскалация, RCE и т.п.)
   - Возможные способы смягчения

(Опционально) Используйте PGP для шифрования (добавьте сюда публичный ключ).

### Сроки реакции (ориентиры)
| Критичность | Примеры | Подтверждение | План исправления | Выпуск |
| ----------- | ------- | ------------- | ---------------- | ------ |
| Критическая | RCE, утечка паролей | ≤ 24 ч | ≤ 48 ч | ≤ 7 дней |
| Высокая     | Эскалация привилегий | ≤ 48 ч | ≤ 5 дней | ≤ 14 дней |
| Средняя     | Логическое обходное действие | ≤ 3 дня | ≤ 10 дней | ≤ 30 дней |
| Низкая      | Минорная информация | ≤ 5 дней | По возможности | В очереди |

## Процесс
1. Получение отчёта → подтверждение.
2. Репродукция → присвоение критичности.
3. Разработка и проверка фикса.
4. Подготовка релиза / патча.
5. Публикация релиза + краткий отчёт (без эксплуатационных деталей).
6. Благодарность исследователю (если согласовано).

## Публичное раскрытие
Публикуется после выхода патча. Эксплойт-код (если уместно) откладывается минимум на 14 дней для высоких / критических уязвимостей.

## Что не считается уязвимостью
- Отсутствие rate limiting без эксплуатации.
- Небезопасные пользовательские пароли (со стороны пользователя).
- Отсутствие HTTP security заголовков в dev-режиме (DEBUG = True).
- Устаревший браузер без современных механизмов защиты.
- Социальная инженерия без технической цепочки.

## Рекомендации по безопасности
- Используйте актуальные версии зависимостей.
- Настройте HTTPS (TLS).
- Применяйте переменные окружения для секретов.
- Регулярно обновляйте Django / DRF.
- Включите SECURITY_MIDDLEWARE и настройки CSP (при необходимости).
